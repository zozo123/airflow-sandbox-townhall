01 · Why now

Code nobody wrote is entering our DAGs

“Airflow lacks native sandboxing mechanisms… DAG authors can execute arbitrary code on workers.” — Airflow security model docs. Mitigation: code review.

That model assumes a human wrote and reviewed every task. With the common.ai provider, agents now generate task code at runtime — and code_mode’s run_code executes it in-process on the worker, with the worker’s credentials. Untrusted by construction; reviewed by no one.

03 · The trap

Today you choose:
fast-and-shared or isolated-and-slow

fast & shared

Celery-style workers start tasks instantly — into a process that shares everything with every other task on the node.

isolated & slow

Pod-per-task and Fargate give a real wall — and charge seconds-to-minutes of cold start for every single try.

No Airflow executor targets ephemeral cloud sandboxes.
That’s the gap issue #68845 names.

04 · What the agent platforms figured out

Per-task cold start, on a log scale

≈ 4 orders of magnitude faster — at a stronger isolation boundary.

Firecracker: AWS / NSDI ’20 (~125 ms boot, <5 MiB overhead, 150 VMs/s/host) · E2B docs · Firecracker snapshot docs · Airflow amazon-provider docs

05 · The mechanism

Snapshots replaced warm pools.
Forks replaced setup.

  • Pay for warm state once, reuse it N times. A “golden” parent runs pip install, imports, cache warmup — every task try forks from it, already hot.
  • Modal: import torch — ~5 s cold → ~1.05 s from a memory snapshot.
  • Cognition (Devin): 20 GB machine snapshot — ~30 min → ~200 ms with blockdiff.
  • Morph: fork a running VM — processes, memory, in-flight state — in < 250 ms.
  • Copy-on-write economics: N forks ≈ 1 parent’s memory + deltas. One-sandbox-per-task becomes affordable.

modal.com/blog/mem-snapshots · cognition.ai blockdiff · morph.so/blog/infinibranch

06 · The proposal — PR #68847

One ephemeral sandbox per task try

Scheduler

Queues task tries exactly as today. No core changes — rides AIP-51’s public executor interface.

→

SandboxExecutor

Creates a sandbox per try, uploads the workload, then only reconciles exit state via a polling watcher.

→

MicroVM sandbox

Task SDK supervisor runs inside: heartbeats + logs go straight to the api-server — EdgeExecutor topology, not a DB-coupled watcher.

→

Destroy

Task ends, sandbox dies. Nothing to clean, nothing to leak into the next try.

Plus SandboxOperator and @task.sandbox for per-task opt-in — mix executors per deployment (AIP-61) and route only untrusted or heavy tasks to sandboxes.

08 · What it looks like

Same DAG, different blast radius

from airflow.sdk import dag, task

@dag
def nightly_train():

    @task.sandbox(
        image="python:3.12-slim",
        cpu=2, memory_mb=2048,
    )
    def train():
        # returns the command the sandbox runs
        return "python train.py --epochs 3"

    train()
# airflow.cfg
[core]
executor = SandboxExecutor

[sandbox]
provider = e2b
poll_interval = 5
creation_batch_size = 8

@task.sandbox mirrors @task.bash: the callable returns the shell command that runs inside the ephemeral sandbox. Requires remote logging — logs ship from the sandbox, not the worker.

10 · Honest limits — and what you can do today

What a sandbox doesn’t fix

still open
  • DAG processor & Triggerer still hold DB credentials
  • No per-task least-privilege secrets scoping yet — JWT identity enables it; ACLs are future work
  • Snapshot sharp edges: TCP connections die, cloned entropy needs VMGenID, GPU state largely unsnapshottable
today, zero airflow changes
  • gVisor / Kata via runtimeClassName in a KubernetesExecutor pod_override
  • gVisor: no KVM needed, ~5–20% syscall overhead — GKE Sandbox is the managed flavor
  • Route only sensitive tasks there with multiple executors (AIP-61)

This is proposal-stage engineering — the caveats are part of the design discussion, not footnotes.
